## Notes

CI checksums don't cover the locally-built DMG — appending it to SHA256SUMS is an explicit runbook step, easy to forget.

Initial APNS/iOS self-hosting research (2026-06-10): there does not appear to be a native iOS background/lock-screen push path that bypasses APNS. This only affects proper mobile push: an iOS self-hoster can still use HTTP refresh and WebSocket updates while the app is running. Third-party push services are wrappers around APNS for iOS and still require an Apple APNS key/certificate for the app bundle. The docs should present two honest shapes:

- Official Nod iOS app + project-operated push relay: avoids sharing project APNS keys, but is not fully self-hosted because push delivery depends on the project relay.
- True self-hosted iOS: the operator needs an Apple Developer Program account, their own bundle ID/App ID with push enabled, their own APNS auth key or certificate, a production TestFlight/App Store build under their team, and matching `NOD_APNS_DIRECT_*` or relay settings.

Privacy/deployment narrative (2026-06-10): v1 is a self-hostable ownership release, not yet a privacy-preserving hosted service. The server admin currently has broad visibility into requests, decisions, and logs. Future privacy modes can support more centralized deployment without requiring the operator to read message contents:

- Private push relay: APNS relay sends generic/opaque notifications and does not see request text; clients fetch details over authenticated server channels.
- Content-private server: request bodies/options are encrypted for recipient devices so the server stores and routes ciphertext while still seeing metadata such as recipients, channels, timestamps, and delivery state.
