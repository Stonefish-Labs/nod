## Goal

For multi-recipient requests, every surface a client can read agrees on the canonical `request_digest`, so a client that recomputes/signs the digest it was shown always passes server verification.

## Acceptance Criteria

- [x] HTTP device view and WS sync envelope expose the same `request_digest` for multi-recipient requests
- [x] A signed decision built from the device-view digest verifies for a request with 2+ recipients
- [x] Regression test with two users/devices pinning the agreement (extend `tests/e2e_smoke.rs` or the API suite)

## Notes

Found by the T-003 review (pre-existing, not introduced there): the HTTP device view computes `request_digest` AFTER per-user recipient filtering (`src/db/requests/read.rs:159` then `to_wire`), while the WS envelope and server-side signature verification use the full-recipient digest (`sync_envelope_for_user` rewrites the recipients JSON but not the digest; `record_decision` verifies via `get_request`). With a single recipient the views coincide, which is why the current smoke can't catch it. Decide the canonical rule (digest binds to the full immutable snapshot — likely the WS/verify side is right and the device view should carry the unfiltered digest) and pin it.

Severity: real correctness bug for multi-user channels; invisible in single-user deployments.

## Progress

2026-06-10: Investigation found the digest bug PLUS a second, worse one with the same root cause: db::record_decision returned the actor's projection and the service layer built the shared-resolution "resolved" broadcast from it — sync::request targets request.recipients (filtered to the actor), so other recipients never received the resolved event; callbacks and audit also saw the projection.

Fix (d138e6c + review follow-up): DecisionRequest carries a serde-skipped canonical_digest stamped by request_for_user before filtering; to_wire/views prefer it; record_decision returns canonical (fanout/callback/audit canonical, actor response projected at the service edge). Regression: API test (device-view digest == canonical + signature verifies, 2 recipients — verified failing pre-fix via stash) and smoke watcher device (created + resolved envelopes reach the non-acting recipient).

Review (fresh-context): LGTM with notes → applied (1) signing.rs verification digest now via to_wire (projection-proof), (2) views.rs avoids a full wire clone, (3) nod-proto doc notes the digest is server-attested for multi-recipient views (not recomputable from a filtered view). Noted, not actioned: non-transactional response re-read (pre-existing shape, ms window, worst case 404-after-success); per-user resolved envelope targeting confirmed correct including the last-decision flip (every recipient already holds their own decision). Reviewer also confirmed no other projected-model path feeds digest/broadcast/callback/audit/push.
