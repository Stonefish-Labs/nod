## Goal

For multi-recipient requests, every surface a client can read agrees on the canonical `request_digest`, so a client that recomputes/signs the digest it was shown always passes server verification.

## Acceptance Criteria

- [ ] HTTP device view and WS sync envelope expose the same `request_digest` for multi-recipient requests
- [ ] A signed decision built from the device-view digest verifies for a request with 2+ recipients
- [ ] Regression test with two users/devices pinning the agreement (extend `tests/e2e_smoke.rs` or the API suite)

## Notes

Found by the T-003 review (pre-existing, not introduced there): the HTTP device view computes `request_digest` AFTER per-user recipient filtering (`src/db/requests/read.rs:159` then `to_wire`), while the WS envelope and server-side signature verification use the full-recipient digest (`sync_envelope_for_user` rewrites the recipients JSON but not the digest; `record_decision` verifies via `get_request`). With a single recipient the views coincide, which is why the current smoke can't catch it. Decide the canonical rule (digest binds to the full immutable snapshot — likely the WS/verify side is right and the device view should carry the unfiltered digest) and pin it.

Severity: real correctness bug for multi-user channels; invisible in single-user deployments.
